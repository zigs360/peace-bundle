# Admin Cash-Out / Wallet-to-Bank Withdrawal — Current Business Logic (Codebase Review)

This document describes the current wallet-to-bank withdrawal and transfer mechanisms in the backend, and specifically evaluates whether an administrator can “cash out total revenue” generated by the system. It also lists validation rules, fee handling, security controls, audit trails, and major gaps against financial best practices (PCI DSS / AML / KYC).

## 1) Executive Summary

### What exists today
- **User wallet → bank transfer (SMEPlug transfer)**: Implemented via `POST /api/transfer/send` and debits the caller’s wallet before initiating a bank transfer through SMEPlug.
- **User wallet → bank withdrawal (gateway withdrawal)**: An endpoint exists at `POST /api/transactions/withdraw`, but the controller calls `paymentGatewayService.processWithdrawal(...)` which is not implemented in the current `paymentGatewayService`.
- **Admin dashboard revenue stats**: Admin stats compute “revenue” as the sum of completed `Transaction` records where `type = 'debit'` (purchases).

### What does *not* exist today (critical for “admin cash-out total revenue”)
- There is **no system treasury / house wallet** that tracks platform revenue as a withdrawable balance.
- “Total revenue” in admin stats is **an analytics metric**, not a balance that can be withdrawn programmatically.
- There is **no admin cash-out workflow** (maker-checker approval, settlement batching, reconciliation, bank beneficiary allowlist, etc.).

Result: Administrators cannot currently cash out “total revenue” using codebase mechanisms; the only implemented bank transfer path is tied to the **admin user’s personal wallet balance**, not platform revenue.

## 2) Key Concepts: Balances vs Revenue

- **Wallet.balance** (liability): Sum of all users’ stored wallet value. This is money you owe users, not platform revenue.
- **Admin “total_revenue”** (analytics): Sum of `Transaction.amount` for completed `type = 'debit'` records. This represents gross purchase volume; it does not necessarily equal net revenue or cash in bank.
- **Funding fees** (recent change): Funding credits record a flat fee and credit a net amount, but the fee is only recorded in metadata; it is not posted into a dedicated revenue ledger.

## 5) Detailed Execution Flow (Implemented Transfer)

### Flow: `POST /api/transfer/send`
1. **Auth**
   - `protect` verifies JWT and loads user record.
   - No requirement for KYC verified or 2FA enabled.
2. **Request validation**
   - `express-validator` checks bank_code, 10-digit account_number, amount > 0, bank_name, account_name.
3. **Fee calculation**
   - Hard-coded `fee = 50.00`
   - Total deduction = `amount + fee`
4. **DB transaction**
   - Opens `sequelize.transaction()`
5. **Balance & limits**
   - Checks wallet exists.
   - Checks `balance >= totalDeduction` in controller.
   - Then calls `walletService.debit(...)` which also enforces:
     - wallet.status must be `active`
     - daily limit and daily spent
     - sufficient balance
   - Wallet row is locked (`lock: true`) within the transaction.
6. **Provider transfer**
   - Calls SMEPlug `/api/v1/transfer/send` via [smeplugService.sendTransfer](file:///c:/Users/7410/peace%20bundle/backend/services/smeplugService.js)
   - On failure: throws error → DB tx is rolled back, so wallet debit is undone.
7. **Transaction record update**
   - Stores `smeplug_reference` and `smeplug_response` on the transaction and commits.
8. **Response**
   - Returns `transactionId`, `reference`, and `newBalance`.
   - Implementation detail: `newBalance` currently comes from `user.wallet.balance` which is not refreshed after `walletService.debit` updates a separate instance.

## 6) Business Rules & Controls (What is enforced vs missing)

### Enforced (current)
- **AuthN/AuthZ**: JWT auth via `protect`; admin-only routes via `admin`.
- **Rate limiting**: global API and admin rate limiters in [server.js](file:///c:/Users/7410/peace%20bundle/backend/server.js).
- **Transfer input validation**: via `express-validator` on transfer routes.
- **Wallet lock + transactional rollback**: debit happens inside a DB transaction with row lock.
- **Daily wallet spending limit**: enforced in `walletService.debit`.

### Missing / incomplete (gaps)
#### Admin cash-out / revenue settlement
- No treasury ledger or platform balance model.
- No “admin cash-out revenue” endpoint or batch settlement job.
- No reconciliation pipeline for provider status updates.

#### Approval workflows
- Transfers are “single-step”: request immediately debits wallet and calls provider.
- No maker-checker approval or multi-admin authorization for high-risk withdrawals.

#### KYC / AML / risk controls
- Withdrawals/transfers do not enforce:
  - `User.kyc_status === 'verified'`
  - BVN verification checks for payouts
  - risk scoring, AML monitoring, PEP/sanctions screening
- No suspicious activity detection (velocity limits by beneficiary, unusual geography/IP, etc.).

#### Bank account verification
- Resolve endpoint exists (`POST /api/transfer/resolve`), but transfer send endpoint does not enforce:
  - that the account was resolved immediately before transfer,
  - that the resolved name matches KYC name (ownership validation),
  - routing number / bank code correctness beyond presence.

#### Audit trail immutability
- Transactions are mutable and can be updated after creation.
- No append-only audit log model for sensitive actions (bank transfer attempts, admin actions, overrides).

#### Sensitive data handling
- Bank account numbers are stored in transaction metadata and beneficiaries in plaintext (JSONB).
- Logs include account numbers in some contexts (risk of leaking sensitive data to log sinks).

#### Withdrawal endpoint correctness
- `/api/transactions/withdraw` route/controller mismatch (`bankCode` vs `bankName`) and missing gateway implementation.

## 7) Security Review Notes

### SQL injection risks
- Most DB operations use Sequelize models or `sequelize.query` with replacements (safer than concatenation).
- Remaining risks are primarily around:
  - logging untrusted input,
  - storing unvalidated bank data in metadata.

### Authentication / authorization
- Admin-only endpoints are protected by role check only.
- No enforced 2FA for admin actions.
- No fine-grained permissioning (e.g., “finance_admin” vs “support_admin”).

### Rate limiting
- Implemented globally and on `/api/admin`.
- No per-user or per-route tailored rate limits for high-risk endpoints (transfer/withdraw).

### Encryption
- No field-level encryption for sensitive bank account numbers.

## 8) Compliance Notes (Current State vs Expectations)

### PCI DSS
- The system appears not to store raw card PAN/CVV, which reduces PCI scope.
- However, ensure webhooks and logs never store card-related fields from providers.

### AML / KYC
- KYC fields exist on the user model (`kyc_status`, `bvn`, `nin`), but withdrawals/transfers do not enforce them.
- No formal AML monitoring pipeline exists in this codebase.

### Regulatory audit trails
- A true audit trail requires immutable append-only logs for:
  - withdrawal approvals
  - beneficiary changes
  - admin cash-outs
  - failed payout attempts and reversals

## 9) Recommendations (to enable true “Admin cash out total revenue”)

To support “admins cash out total revenue”, implement a treasury and settlement system:
- **TreasuryWallet / LedgerEntry**:
  - Maintain platform balances (revenue, fees, reserves) independent of user balances.
  - Post funding fees into treasury ledger.
- **PayoutRequest model**:
  - States: `created → pending_approval → approved → processing → settled|failed|reversed`
  - Include maker-checker approval and optional 2FA challenge for approval.
- **Beneficiary allowlist**:
  - Encrypted account numbers
  - Mandatory account resolution with ownership match to allowed entity
- **Reconciliation job**:
  - Poll provider status and mark settlement finality
  - Maintain reconciliation reports and discrepancy alerts
- **Security hardening**:
  - Per-route rate limiting for withdrawal endpoints
  - Strict input validation for bank code/name and account name normalization
  - Mask sensitive data in logs
